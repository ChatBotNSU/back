# variable_resolver

Реализуем концепт работы с локальными и глобальными переменными и вариативную типизацию.

---

## Cyber Security

Что будет если назвать переменную `eval("fuck_everything_up")`? Манглинг тут не поможет — поможет только префиксный. Но концепция запускать чужой код у себя кажется сомнительной даже в ограничениях ресурсов: допустим код как-то запустится, но будет выполнять незаконные действия. Кто несёт ответственность? Вопрос юридически решаемый, но стоит держать в голове.

## Dev Notes — Какой функционал должен быть реализуем через чатбота

Три показательных кейса:

1. **Финучёт** — API к внешним событиям, работа с внешним API
2. **Агентская система** — ИИ-ассистент (Claude по API), доступ к MCP-серверам, БД для хранилища
3. **Учёт учеников** — кто заплатил, кто нет, рассылки, отчётность

### Выводы из кейсов

**Бот — это не граф, бот — это проект.** У него много состояний, но иногда он просто должен что-то сделать и перехватить контроль.

Возьмём кейс онлайн-школы. Клиент хочет:
- CRUD над учениками (суммы, даты платежей)
- Быстро отметить оплату
- Рассылать напоминания задолжавшим родителям
- Ежемесячную отчётность

Это уже несколько ботов с разными состояниями в одном проекте. Пример: написал имя ученика просто так — бот предлагает отметить оплату; бот в режиме CRUD — функционал другой; бот написал сам — третий сценарий. Одни пользователи видят одно, другие — другое.

**Вывод:** у проекта должны быть подключаемые внешние сущности — файловые хранилища (собственное, Google Drive), БД (внешняя или наша внутренняя).

## Сабграфы

Пользователь хочет сделать сабграф когда:
- **А.** Хочет переиспользовать кусок логики
- **Б.** Хочет вынести его для читаемости графа

### Что нужно для сабграфа

1. Имя
2. Описание зависимостей от внешних переменных — по сути локальный скоуп

Если пользователь уже работал в графе и решил выделить сабграф, мы строим из него функцию: она зависит от переменных, которые нельзя резолвнуть из локального контекста, и возвращает одно из N выходных значений.

Пример:

```python
x = 0
if x < 10:
    y = 20
else:
    y = 10
send_message(y)
```

Пользователь выносит if-блок → это функция, зависящая от `x`, возвращающая `y`:

```python
def if_part(x):
    if x < 10:
        y = 20
    else:
        y = 10
    return y

x = 0
y = if_part(x)
send_message(y)
```

### Структура сабграфа

Каждый сабграф — нода DAG-а, а не последовательная логика:

```json
{
  "name": "subgraph_name",
  "inputs": ["x"],
  "graph": { "nodes": ["..."] },
  "outputs": ["exit_1", "exit_2", "exit_3"]
}
```

Два списка переменных: что принимает на вход (внешние зависимости) и что определяет после себя. Всё остальное — локально внутри графа.

Вопрос остаётся в визуальном представлении этого функционала.
