# greeter

Тот же сценарий, что в `branching`, но логика классификации вынесена в сабграф `classify`. Каждое входящее сообщение бампит счётчик; сабграф читает счётчик и переписывает `reply` в зависимости от него; главный граф собирает финальный текст.

**Что демонстрирует:**

- `subgraph_call` как способ переиспользовать кусок логики. Поведение by-reference: всё, что объявлено в `inputs` сабграфа, привязывается к переменным caller-а; модификации этих имён внутри сабграфа применяются к caller-у после возврата. Здесь `counter` сабграф не трогает, а `reply` перезаписывает — и caller видит новое значение.
- Локальные переменные сабграфа (`thr`) живут только в его фрейме и пропадают после `subgraph_exit`. В caller-е после возврата лежат только `counter`, `reply`, `user_text` — никакой `thr`.
- Сабграф объявляет именованные `exits` (здесь только `done`), caller обязан сопоставить каждый exit со следующим узлом через `exit_next_nodes`.
- В этой папке две версии сабграфа: вложенная прямо в `chatbot.json` и отдельный `subgraph-classify.json` — содержимое идентично. Это специально, чтобы показать оба способа: библиотечный JSON для `/api/v1/subgraph` и встроенный в боте (engine читает только встроенный).

**Ожидаемое поведение:**

```
user > привет
bot  < Привет, привет! Сообщение #1.0, ты — новенький.
user > как дела
bot  < Привет, как дела! Сообщение #2.0, ты — новенький.
user > и
bot  < Привет, и! Сообщение #3.0, ты — знакомый.
user > здарова
bot  < Привет, здарова! Сообщение #4.0, ты — знакомый.
```
